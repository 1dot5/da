# デザインワークフロー

画面設計・情報設計・視覚階層・状態定義の規約。Pencil (.pen) を使う前提。

## 基本原則

- **UI は意思決定支援**。ユーザーが目的を最短で達成できることが最優先
- **繰り返しはシステム化**。コンポーネント／トークン／スペーシングで一貫性を作る
- **仕様は状態を含んで初めて完成**。normal / loading / empty / error / no-permission / disabled
- **情報階層は色に依存しない**。サイズ・位置・余白・タイポで序列を作る

## 画面設計を始める前の確認事項

1. この画面のゴールは何か？（成功の定義）
2. 主ユーザーと利用文脈は？（デバイス / 利用シーン / 頻度）
3. 一次アクション（Primary Action）は何か？
4. エッジケースは？（empty / error / 権限なし / 低速回線 / 認証切れ）
5. 再利用範囲は？（このページ限り / 横断）
6. Web・モバイル両方で成立するか？

## 情報設計 → 視覚階層 → コンポーネント化

```
1. 情報設計      何を伝えるか、何を先に見せるか、何を段階的に出すか
2. 視覚階層      サイズ・位置・余白・色でどう序列を作るか
3. 状態定義      各要素の normal / loading / empty / error / disabled
4. コンポーネント化  繰り返しを部品化、トークンで揃える
```

## 判断ルール

1. **優先順位を決める**。最初に見せるもの / 後でいいものを分ける
2. **詰め込まない**。段階開示（progressive disclosure）で情報を出す
3. **状態は必ず定義**。normal / loading / empty / error / no-permission
4. **一貫性はルールで強制**。コンポーネント・トークン・スペーシング・タイポ
5. **実装者に曖昧さを残さない**。プロースではなく仕様で書く

## Web / モバイル両対応の判断軸

- **レイアウトは viewport に追従**（SCALE / FILL 意図）。左から切るだけは NG
- **タップターゲット 44×44 以上**（モバイル基準を Web でも守る）
- **Hover に依存しない**。タッチデバイスで Hover は存在しない
- **ナロービューポート 360px で破綻しないか** を必ず確認
- **Primary Action はモバイルでも親指到達圏**（下部 or 右下）を検討

## 出力フォーマット（順番を守る）

1. 画面目的 / 成功基準
2. 情報設計（優先度・構造）
3. コンポーネント提案（責務・props・状態）
4. トークン / スタイル指針（色・余白・タイポ）
5. エッジ状態の仕様（empty / error / loading / no-permission）
6. 次アクション（プロトタイプ → 実装）

## よくある失敗

- エッジ状態を未定義のままにして、実装者が場当たりで埋める
- 視覚の不整合がトークン化されておらずマジックナンバーが増える
- 画面目的が不明瞭で要素が肥大化
- Hover 前提の UI がモバイルで機能しない
- 情報階層が色だけで表現され、色弱ユーザー / 印刷 / モノクロで崩れる
